# Track and Role Model

How skills are categorised by the delivery track they serve, and the persona vocabulary each skill uses.

## The five delivery tracks

Space follows a five-track delivery model: Strategy → Architecture → Discovery → Delivery → Refine. Skills are assigned a primary `track:` value that declares which track they belong to.

```
Strategy > Architecture > Discovery > Delivery > Refine
    ^                                                 |
    +------------------------------------------------+
```

| Track | Key question | Produces |
| --- | --- | --- |
| `strategy` | What should we build next and why? | product.md, roadmap.md, portfolio backlog |
| `architecture` | Is the architecture sound for what we intend to build? | solution.md, tech-stack.md, ADRs, contracts |
| `discovery` | Is this ready for development? | WP design.md, WP backlog.md, metrics.md |
| `delivery` | Is this done to standard? | Code, code review, PR/MR, validation |
| `refine` | What did we learn and what should change? | Retrospectives, metrics reports, refinement sessions |
| `utility` | (routing and orchestration) | Skill index, delivery plans |

### Skill assignments by track

**Strategy:** write-product, write-roadmap, write-backlog (domain/portfolio scope), review-product, review-roadmap, refine-product, refine-roadmap, plan-delivery

**Architecture:** write-solution, write-adr, write-contracts, write-tech-stack, plan-adr, review-adr, review-solution

**Discovery:** write-backlog (work-package scope), write-wp-design, write-metrics, review-backlog, review-design, refine-backlog

**Delivery:** implement, review-code, refactor-code, create-mr, validate

**Refine:** refine-product, refine-roadmap, refine-backlog, refine-docs, refine-solution, write-retrospective, write-metrics-report

**Utility:** space-index

### Cross-track skills

Some skills are primarily on one track but useful on another. Use `also-relevant-to-tracks:` in frontmatter:

```yaml
track: refine
also-relevant-to-tracks:
  - architecture
```

## Role taxonomy

Every skill has a single primary `role:` (what persona does the work) and optionally `also-relevant-to-roles:`.

| Role token | Persona in skill body | Typical responsibilities |
| --- | --- | --- |
| `pm` | Senior Product Manager | Strategy, product docs, backlogs, metrics |
| `architect` | Senior Solution Architect | Architecture, ADRs, contracts, WP design |
| `engineer` | Senior Software Engineer | Implementation, code review, refactoring, PRs |
| `delivery` | Senior Delivery Lead | Roadmaps, backlogs, retrospectives, metrics reports |
| `founder` | (shares `pm` persona) | Portfolio strategy alongside PM |
| `utility` | Skill Router | Index and routing only |

### Role in frontmatter

```yaml
role: pm                     # primary — single value, not a list
also-relevant-to-roles:      # secondary — optional list
  - founder
```

The `role:` field informs role views (generated by `bin/generate-views.js`) and the index. It does not affect routing.

### Persona in skill body

The persona statement is the first sentence of the skill body. It must use a **Senior** prefix for all review-* and write-* skills, and for implementation-side skills:

```markdown
You are a Senior Product Manager drafting a product strategy.

You are a Senior Solution Architect reviewing solution.md for structural soundness.

You are a Senior Software Engineer implementing a story against an approved design.
```

The senior framing signals that the agent should apply genuine judgement, not just fill in a template.

## Role views

Generated Markdown tables filtered by role. Run by `bin/generate-views.js`:

```
packages/skills/views/
  pm.md           # role includes 'pm' or 'founder'
  architect.md    # role includes 'architect'
  engineer.md     # role includes 'engineer'
  delivery.md     # role includes 'delivery'
```

Views are read-only generated files. Never edit them directly — run `pnpm --filter @daddia/skills generate:views` to regenerate.
